# LU01.A01 – Repository und Tasks erstellen

## 1) GitHub-Repository erstellen

**Vorschlag Repository-Name:** `terminumfrage-tool` (oder `mini-doodle`).

### Einstellungen beim Erstellen
- Visibility: `Private` (für Schule/Entwicklung) oder `Public` (Portfolio).
- `Add a README file`: **Ja**
- `.gitignore`: `Node`
- License: optional (z. B. MIT)

### Initiale Ordnerstruktur (Vorschlag)
- `frontend/` – React + TypeScript (Vite)
- `docs/` – Projektdokumentation
- `.github/` – Issue-Templates, Labels, Workflows

---

## 2) Projekt-Board auf GitHub erstellen

Erstelle ein **Project (Board)** mit dem Namen:

**`Terminumfrage-Tool – Umsetzung`**

### Empfohlene Spalten
1. **Backlog**
2. **Ready**
3. **In Progress**
4. **Review/Test**
5. **Done**

### Empfohlene Felder
- **Priority**: High / Medium / Low
- **Type**: Feature / Bug / TechTask / Dokumentation
- **Sprint/Phase**: LU01, LU02, …

---

## 3) Arbeitspakete als GitHub-Issues erfassen

Die folgenden Issues decken die im Projektblatt genannten Bereiche (Must/Should/Could) ab.

## Must-Issues

### #1 Projektgrundlage aufsetzen (Frontend + Basisstruktur)
**Beschreibung:**
- Vite + React + TypeScript initialisieren
- Basisrouting und Seitenstruktur vorbereiten
- Einheitliches Layout (Header, Content, Footer)

**Akzeptanzkriterien:**
- App startet lokal mit `npm run dev`
- Startseite wird angezeigt
- Grundstruktur ist versioniert im Repository

---

### #2 Datenmodell definieren (Umfrage, Optionen, Stimmen)
**Beschreibung:**
- Entitäten definieren: Poll, Option, Vote
- Felder und Beziehungen dokumentieren
- Entscheidung für Backend/DB (Supabase oder Firebase)

**Akzeptanzkriterien:**
- Datenmodell ist dokumentiert (`docs/data-model.md`)
- Beispiel-JSON für Poll inkl. Votes vorhanden

---

### #3 Umfrage erstellen (Titel + Beschreibung + Terminoptionen)
**Beschreibung:**
- Formular für neue Umfrage
- Mehrere Datum/Zeit-Optionen erfassen
- Validierung für Pflichtfelder

**Akzeptanzkriterien:**
- Umfrage kann gespeichert werden
- Mindestens eine Terminoption ist Pflicht
- Fehlerhinweise bei ungültigen Eingaben

---

### #4 Öffentlichen Link pro Umfrage erzeugen
**Beschreibung:**
- Eindeutige URL für jede Umfrage
- Aufruf ohne Login möglich

**Akzeptanzkriterien:**
- Link ist eindeutig
- Umfrage kann über Link geladen werden

---

### #5 Abstimmen mit Name (Ja/Nein/Vielleicht)
**Beschreibung:**
- Teilnehmer geben Namen ein
- Stimme pro Terminoption erfassen
- Mehrfachabstimmung sauber behandeln

**Akzeptanzkriterien:**
- Stimme wird gespeichert und angezeigt
- Gleiches Teilnehmer-Handling ist definiert (z. B. Update statt Duplikat)

---

### #6 Resultate übersichtlich anzeigen
**Beschreibung:**
- Ergebnisansicht pro Terminoption
- Anzahl Ja/Nein/Vielleicht je Option

**Akzeptanzkriterien:**
- Resultate sind direkt nach Abstimmung sichtbar
- Beste Option ist klar erkennbar (mindestens visuell hervorgehoben)

---

## Should-Issues

### #7 Eigene Stimme bearbeiten/löschen
**Beschreibung:**
- Teilnehmer kann seine vorherige Abstimmung korrigieren

**Akzeptanzkriterien:**
- Stimme kann aktualisiert werden
- Optionales Löschen der Stimme möglich

---

### #8 Mobile-friendly UI
**Beschreibung:**
- Responsive Layout für Smartphones
- Formulare und Tabellen mobil nutzbar

**Akzeptanzkriterien:**
- Darstellung auf kleinen Viewports ohne horizontales Scrollen
- Hauptfunktionen mobil vollständig nutzbar

---

## Could-Issues

### #9 „Bester Termin“ automatisch markieren
**Beschreibung:**
- Algorithmus zur Bestimmung der besten Option

**Akzeptanzkriterien:**
- Beste Option wird eindeutig hervorgehoben
- Gleichstände werden sinnvoll behandelt

---

### #10 Dark Mode
**Beschreibung:**
- Umschaltbares helles/dunkles Theme

**Akzeptanzkriterien:**
- Theme kann gewechselt werden
- Lesbarkeit bleibt in beiden Modi gut

---

## 5) Priorisierung für den Start

1. #1 Projektgrundlage aufsetzen
2. #2 Datenmodell definieren
3. #3 Umfrage erstellen
4. #4 Öffentlichen Link erzeugen
5. #5 Abstimmen mit Name
6. #6 Resultate anzeigen

Damit ist ein **MVP** erreichbar, der die zentralen Must-Anforderungen erfüllt.

---

## 6) Definition of Done (DoD) für jedes Issue

Ein Issue gilt als abgeschlossen, wenn:
- Code im Repository committed ist
- Funktion lokal getestet wurde
- Akzeptanzkriterien erfüllt sind
- Ticket im Board auf `Done` verschoben wurde
